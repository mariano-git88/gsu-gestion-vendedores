# CLAUDE.md — Dashboard Gestión de Vendedores (GSU)

## Qué es este proyecto
Web app en Streamlit para las reuniones semanales del Jefe de Ventas de GSU con su equipo comercial. Muestra métricas de venta por vendedor: semana corriente, mes acumulado, y cobertura de clientes sobre cartera
asignada.

## Mi rol
Soy el owner técnico. Construyo y mantengo la app. El usuario final en la reunión es el Jefe de Ventas de GSU.

## Contexto operativo
- Uso: una vez por semana, en la reunión comercial de GSU.
- Modelo de datos: **dos fuentes posibles**, coexistentes en la sidebar.
  - **Modo primario:** pull automático desde la API REST de Contabilium
    (`https://rest.contabilium.com.uy`). El usuario elige mes y rango de
    la semana y toca "Sincronizar".
  - **Modo Manual Secundario (fallback):** upload de las 5 planillas xlsx,
    igual que antes. Vive en un expander colapsado debajo del modo API.
  - Ver entrada 2026-04-17 en `_learning/decisions.md` para el racional
    de la coexistencia.
- No hay persistencia entre sesiones: todo se procesa en memoria y se
  descarta al cerrar el navegador.
- Quién opera la app: el Jefe de Ventas (o Mariano) en la reunión.

## Stack técnico
- Python 3.11+
- Streamlit (UI y deploy)
- pandas + openpyxl (lectura de xlsx)
- requests (cliente HTTP para la API de Contabilium)
- Deploy: Streamlit Community Cloud desde repo privado de GitHub

  `client_id` (email admin) + `client_secret` (API Key). Valores en
  `.streamlit/secrets.toml` bajo `contabilium_client_id` y
  `contabilium_client_secret`.

## Fuentes de datos

El dashboard acepta datos desde dos fuentes intercambiables. Ambas
producen DataFrames con el **mismo schema interno**, así el resto del
pipeline (`transforms.py`, `metrics.py`, `views/`, `exports.py`) no
distingue origen.

### Modo Primario — API de Contabilium (`api_loader.py`)

- **URL base:** `https://rest.contabilium.com.uy` (UY). OJO: la API
  de Argentina vive en `rest.contabilium.com` (sin `.uy`) — no
  confundir, las URLs de documentación pública muestran el dominio AR.
- **Auth:** OAuth2 `client_credentials` contra `POST /token`. Token
  Bearer de 24h (86399s). Constantes en `api_loader.py`:
  - `BASE_URL`, `USER_AGENT`, `DEFAULT_TIMEOUT`, `MAX_RETRIES`.
- **Credenciales:** en `.streamlit/secrets.toml`:
  - `contabilium_client_id` — email admin de la cuenta.
  - `contabilium_client_secret` — API Key (obtenible en Contabilium
    → Mi Cuenta → Datos de mi empresa → API).
  - Requiere plan Full o superior.
- **Endpoints utilizados:**
  - `GET /api/clientes/search` — maestro de clientes, paginado.
  - `GET /api/conceptos/search` — maestro de productos Y combos;
    se filtra por `Tipo` (`"Producto"` / `"Combo"`).
  - `GET /api/comprobantes/search?fechaDesde=&fechaHasta=` — listado
    paginado de comprobantes del rango.
  - `GET /api/comprobantes/?id={}` — detalle de un comprobante con
    sus `Items` y su `IDVendedor` (ambos NO vienen en el listado).
    **Ojo: ID va como query param, no como path segment.**
- **Paginación:** parámetro `page=N`. El response trae `Items`,
  `TotalItems` (total global) y `TotalPage` (contraintuitivamente,
  es el tamaño de página = 50, no la cantidad de páginas). Se
  recorre hasta que `len(acumulado) >= TotalItems` o hasta página
  vacía.
- **Mapeo de campos API → schema interno:**

| Interno | API | Endpoint |
|---|---|---|
| `documento` | `NroDoc` | clientes |
| `razon_social` | `RazonSocial` | clientes (maestro — canónica) |
| `vendedor` (clientes) | `IdUsuarioAdicional` → email via `VENDEDORES` | clientes |
| `vendedor` (fc) | `IDVendedor` → email via `VENDEDORES` | comprobantes GetById |
| `fecha` | `FechaEmision` | comprobantes listado |
| `tipo` | `TipoFc` | comprobantes listado |
| `moneda` | `IDMoneda` → `"UYU"` (794) o `"MONEDA_<id>"` | comprobantes listado |
| `sku` | `Codigo` | comprobantes GetById `Items[].Codigo` |
| `producto` | `Concepto` | comprobantes GetById `Items[].Concepto` |
| `unidades` | `Cantidad` × signo del tipo | comprobantes GetById |
| `monto` | `PrecioUnitario × Cantidad × (1 − Bonif/100)` × signo | comprobantes GetById |
| `sku` (productos) | `Codigo` | conceptos |
| `nombre` (productos) | `Nombre` | conceptos |
| `sub_rubro` | `IdSubrubro` → código via `SUBRUBROS` | conceptos |
| `rubro` | `IdRubro` → nombre via `RUBROS` | conceptos |

- **Signo del monto/unidades** en facturación: las NCF vienen con
  `Cantidad` y `PrecioUnitario` positivos en los items, pero el
  comprobante totaliza negativo. `load_fc_api` aplica `-1`
  manualmente cuando `TipoFc ∈ {NCF, NCT, NCE}`
  (constante `TIPOS_NEGATIVOS` en `api_loader.py`).
  Ver entrada 2026-04-17 en `decisions.md` para la validación
  empírica.
- **Concurrencia del N+1 en facturación:** el listado devuelve ~1000
  comprobantes sin items. Para cada uno hay que llamar
  `GET /api/comprobantes/?id={}` para obtener los items y el
  `IDVendedor`. Se hace con `ThreadPoolExecutor(max_workers=10)` →
  ~60s end-to-end.
- **Mappings ID → valor humano** (archivos del repo):
  - `vendedores.py`: `VENDEDORES: dict[int, str]` +
    `VENDEDORES_OP_EXCLUIDOS: frozenset[int]`.
  - `subrubros.py`: `SUBRUBROS: dict[int, str]` + `RUBROS: dict[int, str]`.
  - Se regeneran cruzando xlsx vs API con el script
    `_exploracion-api-contabilium/derivar_mappings.py`. Actualizar
    cuando se dé de alta un vendedor nuevo o un sub-rubro nuevo.
- **Self-test:** `python api_loader.py` corre parseos (offline) y si
  hay credenciales locales en `secrets.toml`, valida auth +
  paginación + los 3 loaders de maestros + `load_fc_api` sobre
  marzo 2026.
- **Rangos sincronizados:** el botón "Sincronizar" de la sidebar
  pullea **5 rangos temporales** en serie (los 3 principales + 2
  comparativos):
  - **Mes**: selector de mes (default = mes actual). Si el mes
    seleccionado es el mes en curso, la `fecha_hasta` se recorta a
    **hoy** (no al último día del mes). Ver entrada 2026-04-18 en
    `decisions.md`.
  - **Semana**: rango arbitrario (default = lunes de esta semana → hoy).
  - **Trimestre**: ventana móvil de **3 meses consecutivos** definida
    por un "mes final" elegible. Default = mes actual como final.
    Ejemplo: mes final = abril 2026 → ventana = feb+mar+abr 2026. Si
    el mes final es el mes en curso, también se recorta a hoy. Se usa
    principalmente en la tab Cobertura. En Modo Manual Secundario no
    se sincroniza trimestre — esa opción solo aparece bajo modo API.
    Ver entrada 2026-04-18 en `decisions.md`.
  - **Mes anterior** (comparativo): mismo rango del mes anterior,
    recortado al mismo día que el rango del Mes actual para que la
    comparación sea apples-to-apples. Ejemplo: si el Mes actual es
    abril 2026 (al día 18), el Mes anterior va `2026-03-01 → 2026-03-18`.
    Solo se usa para el delta MoM en el Resumen. Cache TTL 24h.
  - **Mismo mes año pasado (YoY)**: análogo al anterior pero un año
    atrás. Ejemplo: abril 2026 al 18 → `2025-04-01 → 2025-04-18`.
    Cache TTL 24h.
  - **Por qué TTL 24h en los comparativos**: son meses ya cerrados,
    no cambian entre syncs consecutivos. TTL más largo amortiza el
    costo del N+1 (se paga 1 vez al día en lugar de 1 vez por hora).
  - **Histórico 12 meses (opt-in)**: botón separado en la sidebar
    **"Cargar histórico (12 meses)"** — pullea un rango grande
    (primer día del mes de hace 12 meses → hoy) con TTL 24h. Primer
    pull ~11-18 min; siguientes 24 h usan caché. Habilita las
    features de clientes dormidos, nuevos, retención y frecuencia
    (ver decisión 2026-04-18 Sprint 3). Si no se cargó, las 4
    sub-secciones muestran un aviso "cargá el histórico" y no rompen.
  - Los 3 rangos principales (Mes / Semana / Trimestre) cachean
    independientemente con `@st.cache_data(ttl=3600)`.
- **Timeout global del sync**: 10 minutos (constante
  `SYNC_TIMEOUT_SEC = 600` en el handler del botón). Si se excede,
  se muestra el mensaje **"Lamentablemente Contabilium está caído.
  Por favor probá nuevamente más tarde o utilizá la opción de carga
  manual"**. Mismo mensaje se usa para `ApiError` (fallos de red o
  del servidor). `AuthError` mantiene su mensaje específico de
  credenciales. Ver entrada 2026-04-18 en `decisions.md`.
- **Botón "Resync forzado"**: debajo del sincronizar normal. Limpia
  los caches de `_api_sync_maestros` y `_api_sync_fc` para forzar
  un pull fresco (bypass del TTL de 1h). Útil cuando se acaba de
  emitir una factura y se quiere ver reflejada al instante.

Fallback. Se accede desde el expander colapsado "Modo Manual

reglas, mismo schema resultante — sin cambios respecto al pipeline

Todas vienen de Contabilium con muchas columnas de ruido. Solo las

  - `Tipo` (FAC o NCF — ver reglas de filtrado abajo)
  - `Moneda` (debe ser UYU)

  - `Producto` (nombre del producto, para display)
  - `Cantidad` (unidades — puede ser negativa en NCF)
  - `Subo Total Bonif` (monto neto con bonificación, SIN IVA — ESTA

### 2. `fc_mensual.xlsx` — Facturación del mes en curso
Misma hoja (`Comprobantes`) y misma estructura que fc_semanal.xlsx.

### 3. `clientes.xlsx` — Cartera asignada a cada vendedor
- Hoja: `Clientes`
- Columnas relevantes:
  - `Documento` (RUT — LLAVE CANÓNICA)
  - `Razon Social` (fuente de verdad para display en la UI)
  - `Vendedor Asignado` (email, a quién está asignado el cliente)

Es el universo contra el cual se calcula cobertura. ~986 clientes.

### 4. `productos.xlsx` — Maestro de productos
- Hoja: `Productos`
- Columnas relevantes:
  - `SKU` (llave — matchea contra `Codigo` de facturación)
  - `Nombre`
  - `Sub Rubro` (código corto, ej: 'A', 'BA', 'PM')
  - `Rubro` (categoría superior, opcional para la UI)

### 5. `combos.xlsx` — Maestro de combos
- Hoja: `Combos`
- IMPORTANTE: este archivo NO es una tabla SKU→nombre. Es una lista de materiales: cada fila es un item que compone un combo. Un mismo combo aparece en múltiples filas.
- Columnas relevantes:
  - `SKU Combo` (el SKU del combo, ej: 'COM CDB')
  - `Nombre` (nombre del combo, ej: 'COMBO DE CANDADOS CON EXHIBIDOR')
- El resto de columnas (Codigo, Item, Cantidad) describen el contenido
  del combo y se ignoran — no nos interesa la descomposición.
- Para usar como tabla de lookup: deduplicar por `SKU Combo` y quedarse solo con las columnas SKU Combo y Nombre.

## Reglas de carga y tipos de dato (CRÍTICAS)

### Documento siempre como string
La columna `Documento` debe leerse SIEMPRE como string, nunca como número.
En cada llamada a `pd.read_excel`, usar `dtype={'Documento': str}`. Esto
previene fallas de join cuando una planilla interpreta el documento como
int y otra como str. No aplicar ningún formateo (no quitar puntos, ni
ceros a la izquierda).

### Lectura con hoja explícita
Cada planilla tiene una hoja con nombre específico (`Comprobantes`,
`Clientes`, `Productos`, `Combos`). Al cargar, especificar
`sheet_name=...` explícitamente. No depender del orden ni de "la
primera hoja".

### Validación de columnas al cargar
Para cada planilla, validar al cargar que existan TODAS las columnas
listadas en la sección "Fuentes de datos" arriba. Si alguna falta,
levantar un error claro al usuario indicando qué columna espera la
app y qué columnas encontró. No continuar con columnas faltantes.

### Rename a nombres internos
Después de validar, renombrar las columnas a nombres internos limpios
(snake_case sin espacios) antes de pasar a `transforms.py`. Mapping:

- `Documento` → `documento`
- `Razon Social` → `razon_social`
- `Vendedor` / `Vendedor Asignado` → `vendedor`
- `Codigo` → `sku` (en facturación)
- `SKU` → `sku` (en productos)
- `SKU Combo` → `sku` (en combos, después de deduplicar)
- `Cantidad` → `unidades`
- `Subo Total Bonif` → `monto`
- `Sub Rubro` → `sub_rubro`
- `Nombre` → `nombre`
- `Tipo` → `tipo`
- `Moneda` → `moneda`
- `Fecha` → `fecha`

El resto del código (transforms, metrics, views) trabaja solo con
nombres internos, nunca con los nombres originales del ERP.

## Reglas de filtrado y transformación (CRÍTICAS)

### Exclusión de vendedores operativos
Las filas de facturación cuyo `vendedor` sea uno de los siguientes
NO se contabilizan en NINGUNA métrica del dashboard:

- `OPJESICA@SUPRABOND.COM.UY`
- `OPVALERIA@SUPRABOND.COM.UY`

Son cuentas operativas/administrativas, no representantes comerciales.
Sus operaciones aparecen en la facturación de Contabilium pero no
deben contar para venta, cobertura ni ningún cálculo de performance.

Implementación:
- Es el PRIMER paso del orquestador `prepare_facturacion()`, antes
  incluso del filtrado de NCF.
- El check de "vendedores sin cartera" opera sobre el DataFrame ya
  post-exclusión, así estos dos no aparecen flagueados como huérfanos.
- Las filas excluidas se reportan en el panel de salud como info
  (no warning) para trazabilidad.
- Match exacto por string (case-sensitive). Si el ERP cambia la
  capitalización o el dominio, actualizar la lista en `transforms.py`
  (constante `VENDEDORES_OP_EXCLUIDOS`).

Ver `_learning/decisions.md`, entrada 2026-04-10, para el contexto
completo y las alternativas descartadas.

### Filtrado de notas de crédito
Regla: se conservan las NCF que representan devoluciones de producto,
se descartan las NCF que son descuentos comerciales.

Implementación:
1. Conservar todas las filas con `tipo == 'FAC'`.
2. Conservar las filas con `tipo == 'NCF'` SOLO si `sku` no está vacío
   (no es NaN, no es string vacío). Estas son devoluciones reales —
   tienen un producto asociado, cantidad negativa, monto negativo.
3. Descartar las filas con `tipo == 'NCF'` y `sku` vacío. Estas son
   descuentos comerciales ("10% DTO. COMERCIAL") sin producto asociado.

Las NCF que se conservan netean con las FAC al sumar por SKU o
sub-rubro, lo cual es el comportamiento deseado.

### Validación de moneda
Toda fila debe tener `moneda == 'UYU'`. Si aparece cualquier otra
moneda (USD u otra), levantar un warning en el panel de salud y
EXCLUIR esas filas del cálculo. No intentar conversión automática.

### Join cliente — facturación
Todo join entre facturación (semanal o mensual) y clientes se hace
EXCLUSIVAMENTE por `documento`. Nunca por razón social.

- Tipo: left join de facturación contra clientes.
- Si una fila de facturación tiene `documento` que no matchea en
  clientes → se marca como "CLIENTE NO ENCONTRADO" y se reporta en
  el panel de salud. No se descarta.
- Si una fila tiene `documento` vacío → se marca como "DOCUMENTO
  FALTANTE" y se reporta.

### Razón social canónica
La `razon_social` mostrada en la UI debe venir de `clientes.xlsx`.
La razón social que viene en facturación se descarta tras el join.

### Clasificación de SKU
Cada fila de facturación (después del filtrado de NCF) debe
clasificarse siguiendo este orden:

1. Buscar el `sku` en productos. Si matchea → tomar su `sub_rubro`
   y `nombre`.
2. Si no matchea, buscar en la tabla de combos (ya deduplicada por
   `sku`). Si matchea → asignar `sub_rubro = 'COMBO'` y tomar el
   `nombre` del combo.
3. Si no matchea en ninguno → asignar `sub_rubro = 'SIN ASIGNAR'`
   y reportar el SKU en el panel de salud.

El orden importa: productos primero, combos después.

## Métricas del dashboard

Para cada vendedor, en DOS timeframes paralelos (semana | mes):

### Ventas
- Monto total (suma de `monto`, en UYU)
- **Δ vs mes anterior** (MoM) y **Δ vs mismo mes año pasado** (YoY),
  visibles en el Resumen al lado y debajo del total del mes. Comparan
  apples-to-apples (mismo día del mes cortado en ambos rangos).
  Implementación: `metrics.comparativa_temporal(df_actual, df_prev,
  df_yoy)` devuelve el dict con `monto_actual`, `delta_mom_pct`,
  `delta_yoy_pct`, etc. Si los DFs comparativos no están disponibles
  (Modo Manual o YoY sin histórico), los deltas son `None` y la UI
  muestra solo el número actual.
- Unidades totales (suma de `unidades`)
- **Tickets** (count de comprobantes distintos vía `id_comprobante`)
- **Ticket promedio** = monto_total / tickets
- Desglose de monto por `sub_rubro`
- Desglose de unidades por `sub_rubro`
- Posibilidad de filtrar por SKU específico
- Posibilidad de filtrar por combo específico (SKUs con sub_rubro = 'COMBO')

> **Columna `id_comprobante`**: canónica del schema, presente tanto en
> modo API como en Modo Manual. En API = `Id` del comprobante que
> retorna Contabilium. En Manual = sintético `"vendedor|documento|fecha|tipo"`
> (el xlsx no trae número de comprobante). El proxy subestima levemente
> si un mismo vendedor emite dos comprobantes del mismo tipo al mismo
> cliente el mismo día — caso raro, aceptable como aproximación.

### Inventario (tab dedicada, requiere modo API + histórico 12m)

Tab construida sobre el hallazgo del discovery 2026-04-18: el listado
de conceptos de Contabilium trae `Stock` y `StockMinimo`, y el detalle
de cada combo trae `Items` con su composición. **No se llama a ningún
endpoint adicional** — los campos se agregan al pull de maestros.

**Schema enriquecido del maestro:**

- `df_productos` ahora incluye columnas `stock` (float, unidades) y
  `stock_minimo` (float, umbral que GSU setea en el ERP).
- `df_combos` incluye columna `stock` **derivada de componentes**:
  `stock_combo = floor(min(stock_componente / cantidad_requerida))`.
  Requiere N+1 sobre los combos (9 en GSU ≈ 1 s adicional).

**Funciones de cálculo** (`metrics.py`):

- `ventas_semanales_por_sku(df_hist, hoy)` → por SKU, venta semanal
  promedio bajo 3 cortes temporales:
  - `venta_sem_ultimo_mes` (últimos 30 días / 4.345)
  - `venta_sem_ultimos_3m` (últimos 90 días / 12.86)
  - `venta_sem_mejor_mes` (mejor mes calendario de los últimos 12 / 4.345)
  Valores negativos se clampean a 0.
- `inventario_semanas_stock(df_productos, df_combos, df_hist, hoy)` →
  tabla consolidada con stock, los 3 cortes y **semanas de stock** en
  cada uno. Marca `critico = True` si `semanas_ultimos_3m < 4`.
  Constantes: `SEMANAS_POR_MES = 4.345`, `CRITICIDAD_SEMANAS = 4.0`.

**Reglas de inventario:**

- **Consolidado por SKU** (sin desglose por depósito — Contabilium UY
  no expone endpoint de depósitos y Mariano confirmó que con
  consolidado alcanza).
- **Stock de combos derivado** (no el `Stock` que trae Contabilium
  directamente). Si un combo tiene un componente faltante, el combo
  no se puede armar aunque Contabilium diga otra cosa. Decisión
  conservadora.
- **Umbral crítico**: <4 semanas según el corte de **últimos 3 meses**
  (default). Los otros 2 cortes son informativos, no disparan flag.
- **SKUs sin venta en 3 meses** NO se marcan críticos (no hay demanda
  que proteger). Aparecen con `semanas_... = NA`.
- **Stock = 0 con venta positiva** → `semanas = 0` (ya se acabó, crítico).

**Modo Manual**: la tab degrada con aviso (el xlsx no trae `stock`).

### Cobranzas (tab dedicada, requiere modo API)

Tab nueva construida sobre el hallazgo del discovery 2026-04-18: el
detalle de cada comprobante ya trae `Saldo`, `FechaVencimiento`,
`CondicionVenta` y `Pagos`. **No se llama a ningún endpoint
adicional** — los 4 campos se propagan desde el N+1 existente de
`load_fc_api`.

**Schema enriquecido del DataFrame de facturación (Modo API solamente):**

- `saldo` (float) — monto pendiente del comprobante con IVA.
  0 = cobrado, >0 = adeudado. Replicado en todas las filas del mismo
  comprobante para simplicidad.
- `fecha_vencimiento` (datetime o None) — fecha de vencimiento del
  comprobante. Puede ser None si el ERP no registró plazo.
- `condicion_venta` (string) — ej "30 Cuenta Corriente", "Contado".
- `pagos_count` (int) — cantidad de pagos parciales recibidos.
  Suele ser 0 incluso en comprobantes cobrados (los pagos totales
  no siempre aparecen en ese array; el `Saldo = 0` es la señal
  canónica de cobranza).

**Funciones de cálculo** (`metrics.py`, todas operan colapsando el DF
a una fila por comprobante vía `_deuda_viva_por_comprobante`):

- `aging_por_cliente(df, hoy)` → matriz cliente × bucket
  (Al día / 0-30 / 31-60 / 61-90 / 90+ / Sin vencimiento) con saldo
  en cada celda. Los buckets se calculan sobre
  `hoy - fecha_vencimiento`. Helpers internos `_bucket_aging` y
  constante `BUCKETS_ORDEN`.
- `top_deudores(df, n=20)` → ranking por saldo pendiente sumado por
  cliente. Incluye `comprobantes_pendientes` y `comprobante_mas_viejo`.
- `dias_promedio_deuda_por_vendedor(df, hoy)` → para cada vendedor,
  promedio simple de `hoy - fecha_emision` sobre comprobantes con
  saldo > 0. Proxy de DSO sin necesitar el endpoint de cobranzas.
- `deuda_vencida_vs_corriente(df, hoy)` → dict con total, vencida,
  corriente, pct_vencida, n_vencidos, n_corrientes.

**Reglas de cobranzas:**

- Solo se considera `tipo == FAC`. Las NCF no se cuentan como deuda
  (ya netean contra la FAC original en el ERP — `IdComprobanteAsociado`
  no se usa todavía para tracking fino).
- "Al día" = aún no vencido. No es un bucket de mora.
- "Sin vencimiento" = el ERP no registró plazo. Típicamente son
  contado que ya se cobraron (Saldo = 0, no aparecen) o excepciones.
- El `Saldo` viene **bruto con IVA** (ej: FAC con `ImporteTotalBruto
  = "1.260,00"` tiene `Saldo = "1.537,20"` = 1260×1.22 cuando está
  adeudado). No es comparable con el `monto` neto sin IVA del DF.

**Modo Manual**: la tab degrada con un aviso. Los xlsx no traen saldo
ni vencimiento.

### Análisis longitudinal (requiere histórico 12m)

Cuatro métricas que solo funcionan con el botón "Cargar histórico
(12 meses)" accionado. Las definiciones operativas fueron acordadas
con Mariano 2026-04-18 y están fijas:

- **Cliente dormido** (`metrics.clientes_dormidos`): cliente en
  cartera sin FAC de su vendedor asignado desde hace **más de 90
  días**. Incluye a los que nunca compraron (caso `Nunca`). UI en
  Cobertura, sub-sección al final.
- **Cliente nuevo** (`metrics.clientes_nuevos`): cliente con primera
  FAC en el mes actual Y **sin FAC previas en los 12 meses
  anteriores**. Match estricto. UI en Cobertura, sub-sección al final.
- **Tasa de retención por vendedor** (`metrics.tasa_retencion`):
  A = clientes del vendedor que compraron en el mes calendario de
  `hoy - 6 meses`. B = subset de A que también compró en los últimos
  90 días desde hoy. Retención % = |B ∩ A| / |A| × 100. UI en
  Análisis como bloque 5.
- **Frecuencia de compra por cliente** (`metrics.frecuencia_compra_por_cliente`):
  para clientes con ≥2 FAC propias en el histórico, promedio de
  días entre compras consecutivas. Días por día (no por comprobante:
  si hay múltiples FAC el mismo día, cuentan como una sola compra
  para el cálculo de intervalo). UI en Análisis como bloque 6.

### Cobertura general — métricas adicionales por vendedor

La tabla de cobertura general (tab Cobertura y preview en Resumen) además
de `cobertura_pct` muestra dos columnas derivadas que dan señal de
**calidad** de la cartera de cada vendedor:

- **`concentracion_80`** (columna UI: "Conc. 80%") — cuántos clientes
  concentran el 80% del monto vendido por el vendedor. Cuanto más bajo,
  más dependiente el vendedor de pocos clientes (riesgo). Cálculo: se
  ordenan los clientes FAC del vendedor en su propia cartera por monto
  descendente, se acumula, y se cuenta cuántos se necesitan para superar
  80%. Helper interno `_concentracion_80_por_vendedor` en `metrics.py`.

- **`mix_top3`** (columna UI: "Mix top-3") — string con los 3 sub-rubros
  con mayor participación en la venta del vendedor, formato
  `"A 85% · BA 10% · resto 5%"`. "resto" aparece cuando hay >3 sub-rubros
  con monto positivo. Helper interno `_mix_top3_por_vendedor` en
  `metrics.py`.

Ambas se calculan sobre FAC en cartera propia (match estricto), igual
que `cobertura_pct`. NCF no cuentan.

### Cobertura de clientes

> **Selector de período en la tab Cobertura**: tres opciones —
> **Semana**, **Mes** y **Trimestre**. La opción Trimestre aparece
> solo si se sincronizó desde API (el Modo Manual Secundario no trae
> trimestre). Las métricas de cobertura se calculan sobre el período
> seleccionado.

- Clientes asignados al vendedor (count de `documento` distintos en
  clientes.xlsx donde `vendedor_asignado == vendedor`).
- Clientes con al menos una venta POSITIVA en el período. Definición:
  count de `documento` distintos en facturación donde:
  - `vendedor == <vendedor actual>`
  - existe al menos una fila con `tipo == 'FAC'` (NCF no cuenta para
    cobertura, incluso si hay devoluciones de producto)
- Cobertura general (por vendedor, sin desagregar)
- Cobertura desagregada por sub-rubro
- Cobertura para un SKU específico (selector)
- **Lista de clientes que NO compraron un SKU específico en el mes**
  (selector reutiliza el de cobertura por SKU). Siempre sobre `df_mes`,
  no afectada por el selector de Semana/Mes. Match estricto: si el
  cliente le compró el SKU a otro vendedor distinto al asignado, igual
  aparece (la oportunidad sigue abierta para el asignado).
  Ver `_learning/decisions.md` para el criterio completo.

> **Match estricto en todas las métricas de cobertura**: el matching
> entre facturación y cartera se hace por la tupla `(vendedor, documento)`,
> no por `documento` solo. Una venta cruzada (vendedor A le vende a un
> cliente de B) no cuenta como cobertura para nadie. Ver entrada del
> 2026-04-10 en `_learning/decisions.md`.

### Análisis profundo (tab "Análisis")

Tab dedicada con **cuatro** bloques de exploración estratégica para
identificar oportunidades concretas de venta:

1. **Penetración por sub-rubro** — matriz pivot vendedor × sub-rubro
   con % de cobertura. Identifica huecos de cross-sell agregados.
   Coloreado rojo (huecos) → amarillo → verde (bien cubierto).

2. **Heatmap cliente × sub-rubro** — para un vendedor específico (con
   selector + selector de top N, default 30), matriz con sus mejores
   clientes en filas y los sub-rubros en columnas, con monto vendido
   en cada celda. Las celdas en blanco son los huecos a explorar para
   cada cliente. Escala monocromática de grises.

3. **Pareto de clientes** — distribución 80/20. Identifica el "core
   vital" (los pocos clientes que generan el 80% de la venta) y los
   marca con un highlight crema sutil. Selector "(Todos los vendedores)"
   o vendedor específico.

4. **Patrones temporales** — dos gráficos de barras lado a lado:
   ventas por **día de la semana** (Lun-Dom, en castellano) y ventas
   por **quincena del mes** (1-15 vs 16-fin). Selector de vendedor
   (todos / específico). Útil para detectar concentraciones indebidas:
   un vendedor con todo el monto en la segunda quincena sugiere
   empujón artificial al cierre. Funciones puras en `metrics.py`:
   `ventas_por_dia_semana`, `ventas_por_quincena`.

Las cuatro operan sobre el período seleccionado (selector arriba de la
vista, opciones **Mes / Semana / Trimestre**, default Mes). La opción
Trimestre aparece sólo si el modo API sincronizó el rango trimestral
(ventana móvil de 3 meses — ver sección "Fuentes de datos"). Match
estricto consistente con el resto.

### Export de agenda personal por vendedor

Bloque en la sidebar (después del procesamiento de datos) con un
selector de vendedor y un botón **"Descargar agenda.xlsx"**. Genera
un archivo Excel de 5 hojas para que el vendedor se lleve "su agenda"
de la reunión:

| Hoja | Contenido |
|---|---|
| Resumen | Performance del período + cobertura general + comparativa vs promedio del equipo |
| Mi cartera | Listado completo: documento, razón social, monto mes/semana, ¿compró? |
| Clientes dormidos | Solo los que no compraron este mes |
| Penetración | Fila propia del vendedor de la matriz de penetración |
| Top 80% | Los clientes que conforman el CORE 80% de su Pareto |

Generación cacheada con `@st.cache_data`. Match estricto en todas las
hojas. Ver `exports.py` y la entrada de `decisions.md` del 2026-04-10
sobre el export.
- % cobertura = con_venta / asignados
- Cobertura desagregada por sub_rubro: de los clientes asignados al
  vendedor, cuántos recibieron al menos una venta (FAC) de al menos
  un producto de cada sub_rubro.
- Cobertura por SKU específico (filtrable).

## Panel de salud de datos

El panel vive en su propia tab dedicada (la **7ma**, después de Resumen,
Sub-rubro, Cobertura, Análisis, Cobranzas e Inventario). El usuario
tiene que hacer click para verlo, no aparece junto al dashboard principal. **Excepción**: si hay
alertas estructurales en rojo (vendedores sin cartera o clientes
duplicados), arriba de las tabs aparece un banner discreto avisando
para que no se le pase información crítica al usuario.

Contiene los siguientes contadores. Si todos están en cero → semáforo
verde. Si hay warnings menores → amarillo. Si hay errores estructurales
→ rojo.

Contadores:
- Filas cargadas por planilla.
- Filas con `documento` faltante en facturación (warning).
- Filas con `documento` que no matchea en clientes (warning si menos
  del 5% del total, error si 5% o más).
- Filas con `moneda != 'UYU'` (warning, fueron excluidas).
- SKUs que cayeron en 'SIN ASIGNAR' — listado completo (warning).
- NCF descartadas por ser descuentos comerciales (info, no warning).
- Vendedores en facturación que no aparecen en clientes.xlsx (error).

### Principios de código

- `api_loader.py`, `data_loader.py`, `transforms.py`, `metrics.py` y
  `exports.py` son funciones puras (reciben DataFrames / credenciales /
  sesiones, devuelven DataFrames o BytesIO, sin side effects). No
  importan streamlit.
- Solo `app.py` y `views/` importan streamlit.
- Cache agresivo con `@st.cache_data` y `@st.cache_resource`:
  - `@st.cache_resource` para el `ApiSession` (compartido en el proceso).
  - `@st.cache_data(ttl=3600)` para los pulls de la API (maestros y
    facturación por rango).
  - `@st.cache_data` sin TTL para las funciones de transformación y la
    generación de la agenda — se recalculan solo cuando cambian los
    inputs hasheables.
- Validación estricta de schemas al cargar — error claro al usuario si
  falta algo, nunca fallar silenciosamente.

### Lecciones técnicas a respetar (ver `_learning/errors.md`)

- **NO usar `Styler.background_gradient()`, `.bar()` ni `.text_gradient()`**
  de pandas. Esas funciones requieren `matplotlib` como dependencia oculta
  que NO está en `requirements.txt` (y no tiene sentido agregarla solo
  para gradientes). En su lugar, escribir helpers manuales con
  interpolación lineal en RGB. Hay ejemplos en `views/analisis.py`
  (`_color_for_pct` y `_make_grey_scale`).
- **Usar `Styler.map()`** (NO `Styler.applymap()`). El segundo fue
  removido en pandas 3.x y rompe en producción.
- **El smoke test `streamlit run --headless` solo verifica que el app
  arranque y sirva la pantalla de login.** Las vistas que viven dentro
  de tabs no se renderizan hasta que un usuario navega a ellas, así
  que `streamlit headless` NO detecta errores de runtime que ocurren
  al renderizar el contenido de una tab. Para validar cambios en
  views/, **commit + push + verificación inmediata en producción** es
  más confiable que acumular cambios y validar al final.
- **El entorno local NO es referencia de Streamlit Cloud.** Pueden
  haber dependencias instaladas casualmente que en producción no
  existen, y versiones distintas de pandas/streamlit con APIs
  diferentes. Cuando usar una API ambigua de pandas, elegir siempre
  el nombre más reciente (ver entradas del 2026-04-10 en errors.md).

## Autenticación

- Contraseña única compartida leída de `st.secrets["app_password"]`.
- Si la password ingresada coincide → `st.session_state["authenticated"] = True`
  y se renderiza el resto de la app.
- No hay usuarios individuales, no hay recuperación. Cambio de password
  = editar secrets en Streamlit Cloud y redesplegar.

## Qué NO hacer

- No commitear datos reales de ventas al repo. Las planillas de GSU
  no viven en git bajo ninguna circunstancia.
- No commitear `.streamlit/secrets.toml`. Agregarlo a `.gitignore`.
- No persistir archivos subidos en disco. Procesar en memoria con
  `pd.read_excel(uploaded_file)` y descartar al cerrar la sesión.
- No usar `Razon Social` como llave de join bajo ninguna circunstancia.
- No asumir que los nombres de columnas del ERP son estables entre
  semanas. Siempre validar.
- No intentar convertir USD a UYU automáticamente. Excluir y avisar.
- No contar NCF para la métrica de cobertura de clientes.
- No calcular monto con otra columna que no sea `Subo Total Bonif`
  (en xlsx) / `PrecioUnitario × Cantidad × (1 − Bonif/100) × signo`
  (en API — signo viene de `TIPOS_NEGATIVOS`).
- **No llamar endpoints de escritura de la API de Contabilium**
  (`PUT /UpdateComprobante`, `DEL /DeleteComprobante`,
  `POST /AnularComprobante*`, etc). El `api_loader` solo expone
  funciones que hacen GET. El `client_secret` tiene permisos de
  escritura — tratarlo con el mismo cuidado que una password admin.
- No pegar `contabilium_client_secret` ni el `access_token` en el
  chat, logs o commit. Vivir exclusivamente en
  `.streamlit/secrets.toml` (local, gitignored) y en Streamlit
  Cloud Secrets (producción).
- No modificar los mappings `VENDEDORES` / `SUBRUBROS` / `RUBROS`
  a mano si hay una alternativa automatizable — correr
  `derivar_mappings.py` y pegar el output. Los errores tipográficos
  a mano pueden romper una métrica silenciosamente.

## Personalidad

Pragmático. Esto es un dashboard para una reunión comercial de 60
minutos: las métricas tienen que cargar rápido y ser legibles de un
vistazo. Preferir simplicidad sobre sofisticación. Antes de agregar
un gráfico, filtro o pestaña nueva, preguntar si es necesario.

Cuando aparezca una decisión de diseño no trivial (qué librería de
charting usar, cómo estructurar una vista, cómo manejar un caso borde),
parar y discutirla antes de implementar. No improvisar en silencio.
